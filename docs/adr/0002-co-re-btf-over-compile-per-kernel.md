# 0002 — CO-RE/BTF over compile-per-kernel

- **Status:** Accepted
- **Date:** 2026-06-23
- **Deciders:** K-Henry
- **Milestone:** M0

## Context
eBPF programs read kernel struct fields (e.g. `task_struct → nsproxy → mnt_ns → ns.inum`). Kernel
struct layouts differ across versions, so the program must adapt to whatever kernel it loads on.
Three approaches:

1. **Compile-per-kernel** (BCC-style): ship source + a compiler, rebuild on each target. Requires
   clang + kernel headers at runtime; heavy, fragile, slow.
2. **Hand-rolled struct definitions**: hard-code offsets/layouts. Brittle — breaks on any kernel
   that reorders or resizes a struct, and silently reads wrong memory.
3. **CO-RE (Compile Once – Run Everywhere)** via **BTF**: compile a single object with relocation
   records; the loader rewrites field offsets at **load time** against the target kernel's BTF
   (`/sys/kernel/btf/vmlinux`). aya handles this transparently.

## Decision
We will use **CO-RE/BTF**: one portable object, relocated at load time. Rust bindings to kernel
structs are generated by `aya-tool` into a **committed** `crates/ebpf/src/vmlinux.rs`, regenerated via
`cargo xtask codegen`. **We never hand-roll kernel structs.**

## Consequences
- One compiled object runs across kernel versions — no runtime toolchain, no per-node rebuild.
- **Requires BTF on the target** (`CONFIG_DEBUG_INFO_BTF=y`) — added to the platform support matrix
  and checked by the boot preflight.
- The generated `vmlinux.rs` is committed (reproducible builds; build machines need neither BTF nor
  `bpftool`); it's portable because relocation happens at load time, so generating it on one kernel
  is fine.
- `cargo xtask codegen` has dev-only prerequisites (`bpftool` + `aya-tool` + `bindgen`); normal
  builds don't.

## Alternatives considered
- **Compile-per-kernel** — rejected: runtime clang/headers, large footprint, slow, operationally
  fragile.
- **Hand-rolled structs** — rejected: breaks silently across kernel versions; reading wrong offsets
  in a privileged kernel context is unacceptable.
